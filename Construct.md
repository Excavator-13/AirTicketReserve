# 机票预约平台 MVP 构建需求

## 1. 项目概述

构建一个功能完整的机票预约平台 **MVP**，允许用户搜索航班、预订机票、完成支付、查看订单，并支持退票与改签。  
前后端分离，通过 RESTful API 协作。

## 2. 技术栈

- **前端**：Vue 3 + Vite + Vue Router + Pinia + Axios  
  UI 库建议 Element Plus（不强制，要求界面整洁、响应式）
- **后端**：Django 4.x + Django REST Framework (DRF)  
  认证：JWT（djangorestframework-simplejwt）  
  数据库：开发环境可用 SQLite，需支持后续切换 PostgreSQL  
  异步任务（必须）：订单超时自动取消使用 Django 管理命令 + crontab 定时执行（每 1 分钟扫描一次），项目交付时需包含 crontab 配置说明

## 3. MVP 必须实现的功能模块（按用户故事及验收标准）

### 3.1 用户认证（高优先级，关键）

- **注册**  
  作为新用户，我想要通过手机号或邮箱注册并设置密码，以便拥有平台账户。  
  验收：
  - 手机号/邮箱校验，验证码发送（MVP 可模拟验证码，固定如`123456`）
  - 密码强度检查（至少8位，含字母和数字）
  - 注册后返回成功，并生成JWT token
- **登录**  
  作为已注册用户，我想要通过“账号+密码”或“验证码”登录，以便安全进入平台。  
  验收：
  - 账号密码登录成功返回 access/refresh token
  - 验证码登录（发送及校验，可模拟）
  - 错误提示明确，连续错误5次锁定账号30分钟
- **重置密码**  
  作为用户，我想要通过验证身份后重置密码，以便恢复账户访问。  
  验收：通过注册邮箱/手机号发送重置链接（模拟链接即可），设置新密码后旧密码失效。

### 3.2 航班搜索与浏览（高优先级，关键）

- **条件搜索**  
  作为旅客，我想要输入出发地、目的地、出发日期、乘客人数（成人/儿童/婴儿），以便看到可选航班。  
  验收：
  - 出发地/目的地支持按城市名模糊搜索，自动补全
  - 支持单程/往返切换；往返需选择去程和返程日期
  - 乘客：成人≥1，婴儿数≤成人，儿童>=0
  - 搜索结果展示：航班号、航司、起降时间、机场、飞行时长、经停次数、各舱位最低价
- **筛选与排序**  
  作为旅客，我想要对结果按价格、起飞时间、飞行时长等条件筛选和排序，以便快速找到合适的航班。  
  验收：
  - 支持直飞/中转筛选、航司多选、起飞时段滑块
  - 排序：价格从低到高、起飞时间早晚、飞行时长短长
  - 筛选排序可组合，结果即时刷新
- **航班详情**  
  作为旅客，我想要查看航班退改签规则、行李额度、价格分项，以便全面评估。  
  验收：详情页显示票价、税费、燃油附加分项；退改签费用阶梯；免费行李额；机型等。

### 3.3 预订与订单生成（高优先级，关键）

- **填写乘机人**  
  作为旅客，我想要在选定航班后填写所有乘机人的姓名、证件类型及号码，以便系统生成合规订座。  
  验收：
  - 乘机人数与搜索时一致，可增删，证件号格式校验（中国大陆身份证、护照等基本格式）
  - 支持保存为常用乘机人，下次带出
  - 确认乘机人后生成订单，订单状态为“待支付”，并锁定座位（减少库存），设定30分钟支付倒计时
- **价格汇总与附加服务**  
  作为旅客，我想要在确认页看到总价明细，并可选择保险、选座等附加服务。  
  验收：票价、税费、保险等分项列出，总价动态更新（附加服务MVP可提供2-3项模拟项）
- **订单超时自动取消**  
  作为旅客，我想要若超时未支付，订单自动取消并释放座位。  
  验收：支付倒计时结束后订单状态变为“已取消”，座位数回滚。可通过 Django 管理命令/crontab 模拟实现。

### 3.4 支付结算（高优先级，关键）

- **发起支付**  
  作为旅客，我想要在待支付订单页选择支付方式并完成付款，以便完成购票。  
  验收：
  - 提供“模拟支付”方式（直接返回成功/失败可选，演示成功即可）
  - 支付成功：订单状态→“已支付/出票中”，系统自动生成票号（模拟）并最终变为“已出票”
  - 支付失败：展示错误提示，可重新发起支付
  - 已支付订单防止重复支付
- **支付安全**  
  作为旅客，我想要在支付时金额不被篡改，重复支付有保障。  
  验收：后端校验支付金额与订单金额一致；对已支付订单再次请求支付返回明确提示。

### 3.5 我的订单（高优先级，重要）

- **订单列表**  
  作为旅客，我想要在“我的订单”中按状态和时间查看所有订单，以便管理行程。  
  验收：列表展示订单号、航段、日期、总金额、状态（待支付/已出票/已取消/退票中/已退票等），未支付订单倒计时醒目标识，按状态筛选，支持日期范围搜索。
- **订单详情**  
  作为旅客，我想要查看订单完整信息（乘机人、航班、票号、退改签状态），以便出行前确认。  
  验收：已出票显示电子票号；展示“申请退票”“申请改签”入口（符合条件时）。

### 3.6 退票与改签（高优先级，关键）

- **退票申请**  
  作为旅客，我想要对已出票订单在线申请退票，系统计算可退金额。  
  验收：选择乘客后发起退票，根据购票时的退票规则计算手续费，展示预计退还金额；提交后状态变为“退票审核中”，后台自动审核通过（MVP可简化为即时通过），款项原路退回（模拟），订单状态→“已退票”。
- **改签申请**  
  作为旅客，我想要对已出票订单申请改签，选择新航班并支付差价/收取退款。  
  验收：引导搜索同行程新航班，计算改签费+差价，生成改签单；支付后原票状态“已改签”，新票生成并与原订单关联；若新票价格低，模拟退还差额。
- **进度通知**  
  作为旅客，我想要在退改签关键节点收到通知（MVP可在订单日志中展示，不发真实消息）。

### 3.7 消息通知（中优先级，重要）

- **系统通知**  
  作为用户，我想要收到支付提醒、出票成功、航班变动、订单即将超时等通知。  
  验收：MVP 可采用站内信+订单状态内展示；支付超时前10分钟标记提醒；通知关联订单号，点击可跳转。

## 4. 非功能性要求

- 前端路由守卫：需登录才能访问预订、订单等页面。
- API 采用 RESTful 设计，所有需认证的端点使用 JWT Bearer 令牌。
- 数据校验严格，后端返回统一格式的 JSON（含 code, msg, data）。
- 界面简洁、可用，适配移动端响应式布局。
- 航班数据：请创建 Django 管理命令或 fixture，初始化一些航班数据（至少10个航班，覆盖不同航司、时间、直飞/中转），以便功能演示。

## 5. 需要交付的产出

请生成完整的项目文件结构，包含：

- Django 项目（`backend` 目录），含所有 app、模型、序列化器、视图、URL 配置及管理后台
- Vue 项目（`frontend` 目录），含组件、页面、路由、store、API 封装
- 清晰的 `README.md`，说明环境要求、安装步骤、运行方式
- 数据库迁移文件及初始数据（如航班）

确保所有核心流程可完整运行：注册 → 登录 → 搜索航班 → 查看详情 → 预订（添加乘机人） → 模拟支付 → 订单出票 → 查看订单 → 退票/改签。
所有服务使用 Docker 容器化部署，项目根目录提供 `docker-compose.yml`，一键启动前端、后端及数据库。
